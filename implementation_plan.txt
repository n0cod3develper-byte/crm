# Módulo de Mantenimiento y Órdenes de Trabajo — Plan de Implementación

## Resumen

Implementar un módulo CMMS completo que gestione el ciclo de vida de Órdenes de Trabajo (OT) correctivas y preventivas, asignación de técnicos, descargo automático de inventario, liquidación de servicios y generación de PDF.

**Stack**: Node.js + Express + PostgreSQL (backend) / React + Vite (frontend) — igual al resto del CRM.

---

## Decisiones de arquitectura

### 1. Generación de PDF
**Elegido: Puppeteer** — ya está instalado en `backend/package.json`. Genera un PDF renderizando HTML/CSS, lo que permite reutilizar los mismos estilos del CRM y produce resultados de alta calidad. La alternativa `pdfmake` requeriría instalación adicional y tiene una API de más bajo nivel.

### 2. Consecutivos concurrentes
La tabla `consecutivos` usa `FOR UPDATE SKIP LOCKED` + `SELECT ... FOR UPDATE` en una transacción, garantizando unicidad incluso con peticiones concurrentes sin necesidad de secuencias de Postgres.

### 3. Liquidación atómica
Se aprovecha `withTransaction(fn)` ya definido en `config/database.js`. Todo el proceso (validar stock → descontar inventario → crear liquidación → cambiar estado OT) ocurre en una sola transacción.

## User Review Required

> [!IMPORTANT]
> **Tarifa por hora del técnico**: El modelo de datos almacena `tarifa_hora` en `ot_tecnicos`. ¿Debería leerse automáticamente del registro del empleado (si existe un campo `hourly_rate` en `employees`) o ingresarse manualmente al agregar el técnico a la OT? Actualmente la tabla `employees` **no tiene** ese campo; lo agregaremos como campo opcional.

> [!IMPORTANT]
> **Logo para PDF**: El PDF incluye el logo de la empresa. Actualmente no hay gestión de assets en el backend. El logo se tomará de una ruta estática `backend/src/utils/logo.png`. Si no existe, el PDF se genera sin logo. ¿Desea subir un logo ahora o lo configuramos después?

> [!WARNING]
> **Roles de usuario**: El módulo define 4 niveles (Admin / Supervisor / Técnico / Consulta). El sistema actual solo tiene roles `admin`, `agent`, `viewer`. Se usará la siguiente equivalencia sin modificar la tabla `users`:
> - Admin → `admin`
> - Supervisor de mantenimiento → `agent`
> - Técnico → `agent` (filtrado por vínculo al empleado activo)
> - Consulta → `viewer`
>
> Para roles avanzados en el futuro, recomendamos una tabla `roles` separada.

## Proposed Changes

### 1. Base de datos

#### [NEW] `backend/migrations/006_mantenimiento_module.sql`

Crea 5 tablas nuevas:

| Tabla | Descripción |
|---|---|
| `consecutivos` | Controla el último número de cada serie (OT / CAR) |
| `ordenes_trabajo` | Cabecera de la OT |
| `ot_tecnicos` | Técnicos asignados con tiempos y costos |
| `ot_repuestos_insumos` | Materiales usados |
| `ot_liquidacion` | Resumen financiero de la OT |

También agrega el campo `hourly_rate DECIMAL(12,2)` a la tabla `employees`.

---

### 2. Backend

#### [NEW] `backend/src/modules/mantenimiento/mantenimiento.routes.js`
Registra todos los endpoints bajo `/api/v1/mantenimiento` y extiende `/api/v1/inventario/search`.

#### [NEW] `backend/src/modules/mantenimiento/mantenimiento.repository.js`
Contiene toda la lógica de acceso a datos:
- `findAllOT(filtros)` — listado paginado con joins
- `findOTById(id)` — detalle completo (OT + técnicos + repuestos + liquidación)
- `createOT(data)` — genera consecutivo atómicamente
- `updateOT(id, data)` — solo si estado es ABIERTA/EN_PROCESO
- `softDeleteOT(id)` — anulación (no libera consecutivo)
- `addTecnico / updateTecnico / removeTecnico`
- `addRepuesto / updateRepuesto / removeRepuesto`
- `liquidar(id, data)` — transacción atómica completa
- `getProductividadTecnico / getProductividadEmpresa / getResumenGeneral`
- `searchInventario(q)` — buscador para el formulario

#### [NEW] `backend/src/modules/mantenimiento/mantenimiento.controller.js`
Validaciones, manejo de errores HTTP y delegación al repository.

#### [NEW] `backend/src/utils/pdfGenerator.js`
Genera el PDF de la OT usando Puppeteer con plantilla HTML inline.

#### [MODIFY] `backend/src/app.js`
Agrega import y registro de `mantenimientoRoutes` bajo `/api/v1/mantenimiento`.

#### [MODIFY] `backend/src/modules/inventory/inventory.routes.js`
Agrega endpoint `GET /search` para el buscador del formulario de OT.

---

### 3. Frontend

#### [NEW] `frontend/src/pages/Mantenimiento/MantenimientoPage.jsx`

## Endpoints API (resumen)

### Órdenes de trabajo
| Método | Ruta | Descripción |
|---|---|---|
| GET | `/api/v1/mantenimiento/ot` | Listar con filtros y paginación |
| GET | `/api/v1/mantenimiento/ot/:id` | Detalle completo |
| POST | `/api/v1/mantenimiento/ot` | Crear OT (genera consecutivo) |

| DELETE | `/api/v1/mantenimiento/ot/:id` | Anular (soft delete) |

## Integración con módulos existentes

- **Companies**: el endpoint `/api/v1/empresas/:id/equipos` ya existe en `app.js`.
- **Equipos**: la ficha de equipo mostrará un tab "Historial OTs" — se agrega en `EquiposPage.jsx`.
- **Employees**: se muestra tab "OTs asignadas" en `EmployeesPage.jsx`.
- **Inventory**: los descargos se registran en `inventory_movements` con `type='out'` y referencia al consecutivo de la OT.

## Verification Plan

### Automated testing (manual en dev)
1. Crear OT para empresa "CARGAR S.A.S." → verificar consecutivo `CAR-00001`.
2. Crear OT para otra empresa → verificar consecutivo `OT-00001`.
3. Agregar técnico, registrar salida/regreso → verificar cálculo de mano de obra.
4. Agregar repuesto → verificar que precio/descripción se copian del inventario.
5. Liquidar con stock suficiente → verificar descargo en `inventory_movements`.
6. Liquidar con stock insuficiente → verificar que se bloquea y muestra error.
7. Generar PDF → verificar que se descarga correctamente.
8. Intentar editar OT liquidada → verificar que retorna 403.

### Manual Verification
- Navegar al módulo desde el sidebar → Dashboard con KPIs.
- Crear OT completa end-to-end desde la UI.
- Probar filtros del listado (estado, empresa, tipo, técnico).
- Verificar que equipo se carga dinámicamente al seleccionar empresa.
- Descargar PDF y validar contenido.
